🔷 3️⃣ REGRAS DE COMPOSIÇÃO (DERIVADAS DO DssCard)
Aprendizado consolidado:
3.1 O componente estrutural NÃO instancia filhos DSS automaticamente
Ele:
- Fornece slots
- Fornece layout
- Não cria dependência rígida

3.2 Composição é responsabilidade do consumidor
O componente estrutural:
- Não estiliza filhos internamente
- Não usa ::v-deep para sobrescrever filhos DSS
- Não assume presença de componentes específicos

3.3 Anti-Patterns de Composição (Obrigatório incluir na documentação)
- Usar HTML nativo quando existe versão DSS
- Sobrescrever estilo interno de filho DSS
- Aninhar superfícies excessivamente
- Misturar Quasar bruto com DSS
- Criar dependência estrutural obrigatória

3.4 Comunicação Pai-Filho (NOVO v2.4)
- Prefira Slots (composição via consumidor) em vez de Prop Drilling.
- Se o componente estrutural precisar controlar o estado dos filhos (ex: DssRadioGroup controlando DssRadio), use `provide/inject` tipado, NUNCA manipulação direta de `$parent` ou `$children`.

2️⃣ ARQUITETURA OBRIGATÓRIA (NÍVEL VINCULANTE)

1-structure/

3-variants/

4-output/
- Brands via mixin (sem duplicação)
- States: dark, reduced-motion, contrast, forced-colors, print
- Barrel import obrigatório
- Orquestrador
- <Component>.module.scss com @use das 3 camadas
